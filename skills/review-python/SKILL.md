# review-python

## 目的

Python のコードをレビューし、不具合、型の問題、保守性の低下、不要な複雑さを検出する。

コードの更新は、ユーザーの同意を得た上で実施する。

## レビュー前に実施すること

### 周辺コードを含めて読み込む

- 対象ファイルだけでなく、呼び出し元 / 呼び出し先 / 関連する型定義を、必要に応じて確認する
- 対象コードだけでは判断できない事項を推測で断定しない

## レビュー観点

### module docstring

- module の目的 / 責務を説明する
- file 名や実装から明らかな情報を繰り返さない
- すべての file への形式的な記載は強制しない
- 記載する場合は、次の条件を満たす
  - 50 文字以内
  - 2 文以内の簡潔な文章

### Symbol の配置位置

- 既存プロジェクトの配置規則がある場合は、その規則を優先する
- 関連する Symbol は近くに配置し、処理の流れを追いやすくする
- module の主要な公開 API を先に配置し、補助的な private Symbol はその後に配置する
- 定数、型 alias、Enum などは、原則として利用する class や関数より前に配置する
- 無関係な責務を持つ Symbol が同一 module に混在している場合は、module の分割を検討する
- 見た目を整えることだけを目的とした、必要性のない並べ替えは指摘しない
- 並べ替えによって、decorator、default argument、型 annotation などの実行時参照を壊さないことを確認する

### 命名

- 役割や扱う値を、名前から判断できるものとする
- 略語は、プロジェクト内で一般的なものだけを使用する

**class**

- 名詞または名詞句とする

**関数 / method**

- 動詞または動詞句とする

**真偽値**

- 状態または条件を表す名前とする

### docstring

- `Google Style` を採用する
- summary の記載を必須とする
- summary 以外には、Signature だけでは分からない情報のみを記載する
- 型ヒント記載の型情報を重複して記載しない

### Comment

- 原則として、処理の目的や実装理由などの WHY を書く
- コードから読み取れない情報を書く
- コメントの内容が現在の実装と一致しているか確認する

## コードの更新

- レビュー結果を提示した後、コードを更新してよいかユーザーへ確認する
- レビューで検出した問題を重要度順に提示する
- ユーザーがコードの更新に同意している場合のみ修正する
- レビュー対象外の変更や、必要性のないリファクタリングを行わない
- 更新後は差分を確認し、意図しない変更が含まれていないことを確認する

## 検査

### pyright

project root または workspace に pyrightconfig.json が存在する場合、その配置先で実行する。

```shell
cd <path_to_pyrightconfig_directory>
mise x -- pyright .
```

`pyrightconfig.json` が存在しない場合は、既存のプロジェクト設定を確認し、独自の設定ファイルを追加せずに実行可否を判断する。

### ruff

レビュー対象ファイル / ディレクトリを指定して実行する

```shell
ruff check --config "$HOME/.config/ruff/.ruff.toml" <path_to_target>
```

自動修正は行わず、まず検査結果を確認する。
