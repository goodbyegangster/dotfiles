# review-python

## 目的

Python のコードをレビューし、可読性、保守性、不要な複雑さを評価する。

ユーザーの指示とプロジェクト固有の規約を、この skill の規則より優先する。

## レビュー手順

- 対象ファイルだけでなく、判断に必要な呼び出し元 / 呼び出し先 / 型定義を確認する
- 対象コードだけでは判断できない事項を、推測で断定しない
- 明示的な修正依頼がない場合、レビュー結果を提示してから修正可否を確認する

## レビュー項目

### file / directory の配置位置

新規作成または移動する場合に確認する。

- 対象 directory と周辺 directory の構成を確認する
- 同じ責務を持つ既存ファイルの配置規則に従う
- 呼び出し元と同じ階層であることだけで、file の配置先を決定しない
- 適切な配置先がない場合のみ、新しい directory 作成を検討する
- 独自の分類規則を追加しない
- 配置規則が不明確な場合は、変更前にユーザーへ確認する

### `__init__.py`

`__init__.py` には、原則として以下情報だけを記載する。

- package docstring
- package の公開 API として再公開する import
- `__all__`

### module docstring

- module の目的 / 責務を説明する
- file 名や実装から明らかな情報を繰り返さない
- すべての file への形式的な記載は強制しない
- 2 文以内の簡潔な文章とする

### Symbol の配置位置

- 実行時に評価される依存関係を壊さないことを、最優先とする
- 既存プロジェクトの配置規則がある場合、その規則を優先する
- 特段の規則がない場合、主要な処理の流れを追いやすい順に配置する
- 関連する Symbol を近くに配置する
- 無関係な責務が混在している場合は、module の分割を検討する

### 説明変数 / 説明関数

- 複雑な式、条件、処理には、意図を表す説明変数または private 関数 / method の利用を検討する
- 式の短さより、可読性を優先する
- 一度のみ利用するものでも、可読性が向上する場合は分離する
- 単に行数を増やすだけで、意図が明確にならない分離は行わない

### 命名

- 役割や扱う値を、名前から判断できるものとする
- 略語は、プロジェクト内で一般的なものだけを使用する

#### class

- 名詞または名詞句とする

#### 関数 / method

- 動詞または動詞句とする

#### 真偽値

- 状態または条件を表す名前とする

### 関数 / method

#### 引数

- `self` と `cls` を除く引数が 3 つ以上ある場合、引数を keyword-only 必須とする
- override、protocol、callback、外部APIとの互換性が必要な場合、既存の signature を優先する

#### 戻り値

- 3 つ以上の値を返す場合、tuple として返さず、名前付きの結果型を定義する
- 結果型には、原則として `dataclass` を利用する
  - `dataclass` の docstring には、属性名より意味が類推できない場合 `Attributes` を記述する

#### 副作用

- 計算処理と、I/O や状態変更を可能な限り分離する
- 関数や method が、名前や docstring から予測できない副作用を持っていないか確認する
- 引数や共有状態を、呼び出し側から予測できない形で変更しない
- 副作用は名前から判断できるようにし、必要な補足を docstring に記載する

### docstring

- `Google Style` を採用する
- summary の記載を必須とする
- summary 以外には、Signature だけでは分からない情報のみを記載する
- 型ヒント記載の型情報を重複して記載しない

### Comment

- 原則として、処理の目的や実装理由などの WHY を書く
- コードから読み取れない情報を書く
- コメントの内容が現在の実装と一致しているか確認する

## コードの更新

- ユーザーが明示的に依頼または同意した変更だけを行う
- レビュー対象外の変更や、必要性のないリファクタリングを行わない
- 更新後に差分を確認し、意図しない変更がないことを確認する

## 検査

- コードを更新した場合、更新後に pyright / ruff を実行する
- コードを更新しない場合でも、レビュー判断に必要であれば pyright / ruff を実行する
- プロジェクトに既定の検査コマンドがある場合は、そのコマンドを優先する
- 検査のためだけに、独自の設定ファイルを追加しない
- 検査できない場合は、実行したコマンドと理由を報告し、成功したものとして扱わない

### pyright

`pyrightconfig.json` または `pyproject.toml` の [tool.pyright] を確認し、対象コードに適用される設定の基点で実行する。

```shell
cd <path_to_pyright_config_directory>
mise x -- pyright .
```

### ruff

`pyproject.toml`、`ruff.toml` または `.ruff.toml` を確認し、対象コードに適用される設定の基点で実行する。

```shell
cd <path_to_ruff_config_directory>
ruff check <path_to_target>
```

自動修正は行わず、まず検査結果を確認する。
